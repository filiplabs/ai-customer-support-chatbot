# AI Customer Support Chatbot

A production-deployed customer support chat demo built with Node.js, Express, vanilla JavaScript, and the OpenAI Responses API.

Live demo: https://ai-customer-support-chatbot-9vg7.onrender.com/

## Features

- Short customer-support answers generated by `gpt-5-mini`
- In-session conversation context with bounded server-side history
- Responsive, keyboard-accessible chat interface
- Request validation, rate limiting, concurrency protection, and security headers
- Optional Redis-backed rate limits for multi-instance deployments
- OpenAI request timeout and output-token controls
- Automated backend, browser-behavior, and configuration tests

## Architecture

```text
Browser (public/)
  -> POST /chat
Express application (app.js)
  -> origin and security middleware
  -> request size, rate, and concurrency controls
  -> strict conversation validation and recent-history selection
OpenAI Responses API

server.js
  -> validates environment configuration
  -> connects the optional Redis rate-limit store
  -> starts and gracefully stops the HTTP server
```

Conversation history is held in browser memory for the current page session. The browser sends that history on each request; the server validates all messages and forwards only the latest five completed exchanges plus the current user message.

## Requirements

- Node.js 20 or newer
- An OpenAI API key
- Redis for durable, shared production rate limiting (recommended)

## Local setup

```bash
git clone https://github.com/filiplabs/ai-customer-support-chatbot.git
cd ai-customer-support-chatbot
npm ci
```

Copy `.env.example` to `.env`, set `OPENAI_API_KEY`, and then run:

```bash
npm start
```

Open `http://localhost:3000`.

## Configuration

| Variable | Required | Default | Purpose |
| --- | --- | --- | --- |
| `OPENAI_API_KEY` | Yes | None | Authenticates OpenAI API requests. |
| `PORT` | No | `3000` | HTTP listening port. |
| `NODE_ENV` | No | `development` | Enables the production proxy default and production warnings. |
| `REDIS_URL` | Production recommended | Empty | Enables durable shared rate limiting. Without it, limits are process-local. |
| `TRUST_PROXY` | No | `1` in production | Express trusted proxy hop count or proxy-addr value. Match this to the deployment topology. |
| `ALLOWED_ORIGINS` | No | Empty | Comma-separated origins allowed to call the API cross-origin. Same-origin requests do not need this. |

For Render, configure the API key and Redis URL as environment variables. Verify the proxy hop count for the deployed service before changing `TRUST_PROXY`.

## API

### `POST /chat`

Request:

```json
{
  "messages": [
    { "role": "user", "content": "What is your return policy?" }
  ]
}
```

Success response:

```json
{ "reply": "..." }
```

Error response:

```json
{ "error": "..." }
```

The route and response shapes remain stable. Messages must alternate between `user` and `assistant`, begin and end with `user`, and meet the configured size limits.

### `GET /health`

Returns `{ "status": "ok" }` without calling OpenAI.

## Operational limits

- 10 chat requests per client IP per 15 minutes
- 5 concurrent OpenAI requests per application instance
- 32 KB maximum JSON request body
- 500 characters per user message
- 100 messages and 30,000 content characters per submitted conversation
- Latest 11 messages forwarded to OpenAI
- 300 maximum output tokens
- 30-second OpenAI request timeout with automatic SDK retries disabled

## Remaining production considerations

- Rate limits are not durable unless `REDIS_URL` is configured.
- Conversation history is intentionally not persisted across page reloads.
- The health endpoint reports process health, not an active OpenAI or Redis probe.
- This public demo has no user authentication or customer-specific knowledge base.
